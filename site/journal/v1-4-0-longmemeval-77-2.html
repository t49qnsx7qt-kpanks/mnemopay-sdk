The benchmark

LongMemEval is a benchmark designed for exactly the scenario most memory benchmarks skip: long, multi-session histories with conflicting facts and temporal gaps. The oracle variant tests whether a model retrieves the right answer against a ground-truth answer set, scored by a judge model. We used Sonnet-4 for retrieval and GPT-4o as the judge, across 500 questions spanning single-session, multi-session, and cross-context categories.

The 77.2% overall score is honest — it's not cherry-picked from our best category. The multi-session bucket at 66.9% is where we're genuinely weakest, and that's the one that matters most for agents running across days or weeks. Threading context across gaps where the agent wasn't active is a harder problem than we've fully solved. I want to say that plainly rather than bury it in the marketing copy.

The weakest bucket isn't a footnote. Multi-session retrieval is the category that matters most for production agents. 66.9% means roughly 1 in 3 multi-session queries retrieves the wrong fact. We know why, and it's on the roadmap.

Memory that decays and strengthens

The core memory system in MnemoPay is built on two principles from neuroscience research: Ebbinghaus forgetting curves and Hebbian reinforcement. Memories decay in salience over time unless they're retrieved — that's the forgetting curve. Memories that keep being retrieved get reinforced — that's the Hebbian part.

In practice this means that when an agent recalls memories, it sees a ranked list where recency and retrieval frequency both factor into the score. A memory written six months ago that's been retrieved 40 times ranks higher than a memory written yesterday that's never been touched. A memory written yesterday about something that overrides the old fact ranks higher still, because the decay on the old fact has kicked in.

This is what lets us handle the scenario where "user prefers morning meetings" from three months ago competes with "user now works nights" from last week. The old memory has decayed. The new one surfaces first. The LLM doesn't have to figure out which one to trust from a flat blob — the SDK has already resolved the ranking.

EWMA anomaly detection and canary honeypots

One of the additions in v1.4.0 that I'm most confident about is the behavioral anomaly detection layer. Every agent builds a rolling behavioral baseline using EWMA — exponentially weighted moving average. When the observed behavior on a given payment event deviates past a threshold from that baseline, the SDK flags it. This runs on every payment event, not on a cron.

// simplified EWMA anomaly check — runs on every payment event
const alpha = 0.15;
ewma = alpha * observed + (1 - alpha) * ewma;
const deviation = Math.abs(observed - ewma) / ewma;
if (deviation > ANOMALY_THRESHOLD) {
  await flagAnomaly(agentId, { observed, ewma, deviation });
}

Canary honeypots supplement the EWMA layer. These are deliberate tripwires planted in the memory store — entries that should never appear in retrieval results under normal operation. If they get retrieved, it means something is wrong with the retrieval path, either a bug in memory scoring or something more adversarial. Catching that in the SDK before it reaches the payment decision is the whole point.

Merkle integrity on memory

Every memory write in v1.4.0 is hashed into a Merkle chain. That means forget operations are auditable — the hash of what was forgotten is retained in the chain. You can verify at any point that a memory state hasn't been tampered with. For agents handling budgets or making financial decisions, this matters for the same reason financial systems keep audit logs: you need to be able to reconstruct what the agent knew when it made a decision.

The chain verification is fast enough to run inline. It's not a background job you trigger periodically — every write is integrity-checked at write time.

Payment rails and the portable credit score

The other half of MnemoPay is payments. Stripe, Paystack, and Lightning are all live — real rails, not sandbox mocks. Every payment event generates an HMAC-SHA256-signed receipt. If you need to audit what an agent spent last month, you have a cryptographically verifiable trail.

The portable agent credit score (300-850) is new in v1.4.0. It's built from behavioral signals: payment consistency, anomaly flags, retrieval accuracy over sessions, identity verification status. The score lives in the JWT the SDK issues. Any system that accepts the token can read the score without a separate API call — the agent carries its own reputation across systems.

import { MnemoPay } from '@mnemopay/sdk';

const client = new MnemoPay({ apiKey: process.env.MNEMOPAY_API_KEY });

// store a memory — decay and Merkle hash happen automatically
await client.memory.store(agentId, {
  content: 'user works evenings, async is preferred',
  context: 'session-22',
  importance: 0.9
});

// retrieve with decay-weighted ranking
const recall = await client.memory.recall(agentId, {
  query: 'user schedule preferences',
  limit: 5
});

// charge with HMAC receipt
const payment = await client.payments.charge({
  agentId,
  amount: 4.99,
  currency: 'usd',
  rail: 'stripe',
  memo: 'vector search batch'
});

// read portable credit score
const { score, tier } = await client.identity.getCreditScore(agentId);
// score: 742, tier: 'trusted'
